# 订阅套餐限额推算方法与参考

本文档说明如何根据上游 ChatGPT 席位的实际能力，推算 sub2api 订阅套餐的日/周/月限额（USD）。

如果在你的站点里，一个上游账号通常还要额外绑定住宅 IP 才能稳定工作，那么做套餐定价测算时，成本应按“账号采购成本 + 住宅 IP 成本”统一考虑，而不是只看账号本身价格。

## 管理后台住宅 IP 成本口径

管理控制台 Dashboard 的“套餐定价测算”现在仍然只需要填写一个参数：

- **住宅 IP 价格**：单位为 **美元 / (GB*月)**

现在 Dashboard 直接使用统一的“新住宅 IP 核算方法”参与套餐测算，不再额外展示一组单独的双口径说明子面板。核心思路是：

- 以最近最多 14 天的住宅代理 usage 为样本窗口。
- 优先使用真实代理字节，缺失部分再用“站内 observed bytes 动态校准 + 历史参考值兜底”的 token fallback 补足。
- 先得到样本窗口下的人均月流量，再按“测算用户数 / 样本用户数”投影成目标用户池总月流量。
- 最后再乘住宅 IP 单价和汇率，直接并入套餐测算里的实际成本。

如果管理员已经在 Dashboard 里填过非零住宅 IP 单价，系统会记住最近一次有效输入，后续继续作为默认值使用；不再把某个外部供应商样本价格硬编码成全站默认值。

### 流量估算规则

系统会读取最近最多 14 天的历史 usage，并按下面的优先级估算住宅 IP 流量：

1. 优先使用 `usage_logs` 中已落库的住宅代理流量快照。
2. 如果单条 usage 只有部分真实字节，则把真实请求/响应字节记为 observed，把补足部分记为 token estimate。
3. 如果是历史旧 usage、当时还没有落库代理流量字段，则退回到 token 折算。
4. **实际天数**取样本窗口内最早一次有效 usage 所在自然日到当天的自然日跨度，含首尾，最大 14 天。
5. 月成本按下面公式推算：

```text
月流量 GB = （窗口总流量 GB / 实际天数） × 30
住宅 IP 月成本（USD） = 月流量 GB × 住宅 IP 价格（USD / (GB*月)）
住宅 IP 月成本（CNY） = 住宅 IP 月成本（USD） × 美元/人民币汇率
```

其中美元/人民币汇率继续复用系统现有汇率解析与回退机制。如果最近 14 天没有有效住宅代理 usage，则住宅 IP 成本按 0 处理。

### 当前默认校准值

当前系统会优先用 sub2api 自己最近一段时间的住宅代理 observed bytes 样本，动态反推出更接近本站实际流量结构的 `Bytes/token` 校准值；只有当近期样本不足时，才回退到历史校准参考值 `7.096031857 Bytes/token`。

这意味着：

- 新站点或低流量站点，会先走历史参考兜底值。
- 有足够住宅代理 usage 的站点，会逐步转向“按自己站点真实 payload 结构”动态校准。
- 你提供过的供应商账单样本更适合作为人工对账参考，而不是直接作为线上估算时读取的流量数据来源。

### 后台展示如何理解

在 UI 上：

- 住宅 IP 月成本主卡片会直接展示当前新核算方法得到的目标用户池月流量与月成本。
- 这部分成本会直接参与“达标套餐价格”“保守月成本”“预测月利润”等核心计算。
- 卡片文案会继续保留校准系数、样本天数、涉及用户数和汇率等解释信息，便于理解结果来源。

更完整的对账背景、字段解释和已知边界，请参考 [住宅 IP 流量对账与校准说明](./住宅%20IP%20流量对账与校准说明.md)。

## 核心公式

```
套餐限额 = N 个上游席位 × 单席位 API 等价消耗
```

关键变量：
- **N**：套餐承诺的席位等价数量（如 Standard = 3 席位）
- **单席位消耗**：受 ChatGPT 消息限额和 API 定价共同决定

## 三种推算方法

### 方法一：基于消息限额

```
单席位日均消息数 = 消息限额/3h × 有效使用窗口数
API 等价消耗 = 消息数 × 混合模型加权单价/条
```

以 GPT-5.3 为主力模型的计算：
- 10 小时有效工作日：160 × (10/3) ≈ 533 条/席位/天
- 混合模型加权单价：~$0.025/条（GPT-5.4 + mini + o4-mini 混合）
- 3 席位：1,600 条/天 × $0.025 ≈ **$40/天**
- 月（60% 有效利用率）：**~$720/月**

### 方法二：基于平台实际数据

从 sub2api 管理后台获取分组使用量，计算：

```
人均日消耗 = 分组 7 日总消耗 / 7 / 活跃订阅数
席位系数 = 目标席位数 / 当前人均席位占有数
推算限额 = 人均日消耗 × 席位系数
```

实例（2026-04-16 ~ 2026-04-23 实际数据）：
- GPT_Standard 分组 7 日总消耗：$406.58
- 5 个活跃订阅，人均日消耗：~$15/天
- 9 个账号由 5 人共享 ≈ 1.8 席位/人
- 目标 3 席位，系数 = 3/1.8 = 1.67
- 推算：$15 × 1.67 ≈ **$25/天**，月均 **$750-1,250**

> 注意：此方法依赖实际数据，受当前限额是否触顶影响。如果用户频繁触顶，实际消耗被截断，推算结果会偏低。

### 方法三：基于订阅/API 等价比

```
API 等价价值 = 席位月费 × 等价倍率
```

社区公认等价倍率：
- Plus ($20/月) → API 等价 $300-500/月（15-25x）
- Team ($25-30/月) → API 等价 $400-600/月（13-20x）

3 席位：$1,200-1,800/月

> 注意：这是理论最大值。实际用户不可能 24/7 满载，需打 60-70% 折扣。

## 阶梯定价策略

三档套餐以 Standard 为基准，额度线性递增、价格阶梯折扣：

```
额度倍率:  Standard = 1×,  Premium = 2×,  Ultra = 4×
价格折扣:  Standard = 原价, Premium = 9折,  Ultra = 8折
```

即：
- **Premium 额度** = 2 × Standard 额度，售价 = 2 × Standard 售价 × 0.9
- **Ultra 额度** = 4 × Standard 额度，售价 = 4 × Standard 售价 × 0.8

以 Standard ¥90/月为例：
- Premium: ¥90 × 2 × 0.9 = **¥162/月**
- Ultra: ¥90 × 4 × 0.8 = **¥288/月**

## 推荐限额设定

### 比例约束

取三种推算方法的中位数确定 Standard 基准，再按阶梯倍率外推：

| 比例关系 | 建议值 | 说明 |
|----------|--------|------|
| 日 : 周 | 1 : 6 | 允许工作日集中爆发，周末自然回落 |
| 日 : 月 | 1 : 24 | 兼顾工作日和全天候使用模式 |
| 周 : 月 | 1 : 4 | 恰好 4 周 |

### Standard 基准推导过程（3 Team 席位等价）

Standard 套餐的目标是让用户获得 **3 个 ChatGPT Team 席位** 的等价额度。以下用三种独立方法交叉验证，最终取中位数。

**方法一：基于消息限额推算**

```
GPT-5.3 限额 = 160 条 / 3 小时 / 席位
10 小时工作日消息数 = 160 × (10/3) ≈ 533 条 / 席位 / 天
3 席位合计 = 533 × 3 ≈ 1,600 条 / 天

混合模型加权单价 ≈ $0.025 / 条
  （GPT-5.4 $0.0275/条 × 40% + mini $0.015/条 × 40% + o4-mini $0.005/条 × 20%）

日均消耗 = 1,600 × $0.025 = $40
月均消耗（60% 有效利用率）= $40 × 30 × 0.6 = $720
```

→ 得出：**日 $40，月 $720**

**方法二：基于平台实际数据推算**

```
远程站点实际数据（2026-04-16 ~ 2026-04-23）：
  GPT_Standard 分组 7 日总消耗 = $406.58
  活跃订阅用户 = 5 人
  人均日消耗 = $406.58 / 7 / 5 ≈ $11.6

但此数据受 $30 日限额截断影响（管理员已触顶），实际需求被压制。
剔除截断影响，活跃用户真实日均需求 ≈ $15-20

上游账号数 = 9，由 5 人共享 ≈ 1.8 席位 / 人
目标 3 席位，系数 = 3 / 1.8 = 1.67

推算日均 = $15 × 1.67 ≈ $25
月均 = $25 × 30 = $750（保守）
月均 = $20 × 1.67 × 30 ≈ $1,002（修正截断后）
```

→ 得出：**日 $25-33，月 $750-1,000**

**方法三：基于订阅/API 等价比推算**

```
社区公认 ChatGPT 订阅的 API 等价倍率：
  Plus ($20/月) → API 等价 $300-500/月（15-25x）
  Team ($25-30/月) → API 等价 $400-600/月（13-20x）

3 个 Team 席位的 API 等价 = $400-600 × 3 = $1,200-1,800/月
实际用户不可能 24/7 满载，打 65-70% 折扣：
  月均 = $1,200-1,800 × 0.65 = $780-1,170

日均 = $780-1,170 / 30 = $26-39
```

→ 得出：**日 $26-39，月 $780-1,170**

**三种方法交叉汇总**

| 方法 | 日均范围 | 月均范围 |
|------|----------|----------|
| 消息限额 | $40 | $720 |
| 实际数据 | $25-33 | $750-1,000 |
| API 等价比 | $26-39 | $780-1,170 |
| **交集范围** | **$26-40** | **$720-1,170** |

**最终取值**

取交集上沿并加 25% 缓冲（容纳突发高峰），同时满足 1:6:24 的日周月比例约束：

```
日均 = $40 × 1.25 = $50
周均 = $50 × 6   = $300
月均 = $50 × 24  = $1,200
```

| 维度 | 推荐值 | 计算依据 |
|------|--------|----------|
| 日限额 | **$50** | 交集上沿 $40 × 1.25 缓冲 |
| 周限额 | **$300** | 6× 日限额（工作日集中爆发，周末回落） |
| 月限额 | **$1,200** | 24× 日限额（落在三种方法的交集范围内） |

## 验证清单

设定新限额后，通过管理后台 API 验证：

1. **检查触顶率**：观察活跃用户的 `daily_usage_usd` 是否仍频繁触顶
2. **检查分组总负载**：分组日均消耗不应超过上游账号总承载能力
3. **检查经济性**：人均消耗 × 席位数 × API 单价 + 账号采购成本 + 自动折算的住宅 IP 月成本 ≤ 套餐售价 + 可接受利润率

```bash
# 查看分组使用量汇总
curl -s -H "x-api-key: $REMOTE_ADMIN_API_KEY" \
  "$REMOTE_BASE_URL/api/v1/admin/groups/usage-summary"

# 查看仪表盘分组统计（7天）
curl -s -H "x-api-key: $REMOTE_ADMIN_API_KEY" \
  "$REMOTE_BASE_URL/api/v1/admin/dashboard/groups"

# 查看订阅用户明细（含日/周/月用量）
curl -s -H "x-api-key: $REMOTE_ADMIN_API_KEY" \
  "$REMOTE_BASE_URL/api/v1/admin/groups/6/subscriptions"
```
