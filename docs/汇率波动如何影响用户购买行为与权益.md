# 汇率波动如何影响用户购买行为与权益

本文专门解释 Sub2API 里一个很容易让人混淆的问题：

- 用户是按人民币充值、支付和看余额的
- API 的 usage、quota、rate limit、订阅额度却主要按美元语义建模

这就意味着，汇率波动不会用同一种方式影响所有用户。它对“标准余额按量计费用户”和“订阅套餐用户”的影响明显不同。

如果只记一句话，可以先记这句：

> 在本项目里，汇率波动主要影响“标准余额按量计费”用户的余额消耗速度；对“订阅套餐”的价格、已购有效期和已购额度，影响更间接，也更弱。

---

## 先看系统的基本记账原则

当前项目故意把两套金额语义分开了：

- 用户余额、充值金额、订单支付金额：按 `CNY`
- API usage 成本、quota、rate limit、订阅分组的 `*_usd` 限额：按 `USD`

这样设计的目的不是让系统更复杂，而是避免把“用户花了多少钱”和“上游 API 官方成本是多少”混成一件事。

相关代码与文档可以从这些位置看到：

- `README.md`
- `docs/PAYMENT_CN.md`
- `frontend/src/utils/format.ts`
- `backend/internal/service/usage_billing.go`
- `backend/internal/service/exchange_rate_service.go`

这套语义不是“文档上这么写”，而是前后端代码都在主动执行这套区分。

前端格式化函数里，金额语义被硬编码拆开了：

```ts
export function formatBalanceAmount(amount) {
  return formatCNY(amount)
}

export function formatPaymentAmount(amount) {
  return formatCNY(amount)
}

export function formatUsageCost(amount) {
  return formatUSD(amount)
}

export function formatUsageChargeAmount(amount) {
  return formatCNY(amount)
}
```

也就是说：

- 余额、支付金额默认走 `CNY`
- usage 成本默认走 `USD`
- usage 换算后的实际扣款又单独走 `CNY`

这段设计在 `frontend/src/utils/format.ts` 里非常明确。它直接决定了为什么用户会在同一个页面里同时看到：

- 美元的 `actual_cost`
- 人民币的 `charged_amount_cny`

订阅卡片和购买页也在主动强化这种双币种语义：

- `frontend/src/components/payment/SubscriptionPlanCard.vue` 用 `formatPaymentAmount(plan.price)` 展示套餐价格
- 同一个组件又用 `formatUsageCost(plan.daily_limit_usd)` / `formatUsageCost(plan.monthly_limit_usd)` 展示套餐额度
- `frontend/src/views/user/PaymentView.vue` 也是同样的写法

这就是为什么用户看到的套餐天然会呈现成“人民币价格 + 美元额度”，而不是系统临时算出来的偶然结果。

## 为什么汇率问题在这个项目里一定会出现

因为 Sub2API 不是一个“从头到尾只用一种货币”的系统。

它实际上同时处理两条链路：

- 一条是用户侧支付链路，单位是人民币
- 一条是上游模型成本链路，单位是美元

只要这两条链路同时存在，就一定需要回答一个问题：

“某次 API 使用产生了 `X USD` 的实际成本，那么最终应该从用户的人民币余额里扣多少？”

这就是汇率进入业务的地方。

从源码角度看，汇率真正进入业务的入口主要是两步：

1. `exchangeRateService.ResolveUSDCNYRate(...)`
2. `BuildUsageChargeSnapshot(costUSD, resolvedRate)`

第二步会把：

- `USD` 成本
- 有效汇率
- 汇率来源
- 拉取时间
- 安全边际

一起组装成一个 `UsageChargeSnapshot`。后面无论是 usage 日志，还是实际余额扣减，都是围绕这个快照继续走，而不是每个环节各自再去乱算一遍。

## 标准余额按量计费用户会受到什么影响

这是汇率影响最直接的一类用户。

### 余额本身不会自动变多或变少

用户充值进来的余额是人民币账本。比如用户充值了 `¥100`，这个余额不会因为美元涨跌自动重估成 `¥102` 或 `¥97`。

也就是说：

- 汇率变化不会改写用户钱包里的余额数字
- 汇率变化会影响这笔余额未来能支撑多少次按量调用

之所以会这样，不是因为“产品希望如此”这么简单，而是因为代码里根本没有“汇率变化后批量重估用户余额”的逻辑。

用户余额的变化来自显式的加减操作，而不是来自汇率重算。比如：

- `backend/internal/repository/user_repo.go` 里的 `DeductBalance(...)` 本质上只是 `AddBalance(-amount)`
- `backend/internal/service/payment_order.go` 里创建订阅订单时，订单金额直接来自 `plan.Price`

也就是说，系统会在这些时点改余额：

- 用户充值到账
- 用户购买订阅
- 用户发生一次按量 usage 扣费
- 管理员手工调账

但系统不会在“今天美元涨了”这件事发生时，把所有用户余额扫一遍重新估值。

### 真正会变化的是每次 usage 对人民币余额的扣减额

按量请求在内部先算出 `USD` 成本，再换算成 `CNY` 扣款。

当前链路里已经显式区分：

- `BalanceCostUSD`
- `BalanceCostCNY`
- `FXRateUSDCNY`
- `FXRateSource`

相关代码入口：

- `backend/internal/service/usage_billing.go`
- `backend/internal/service/exchange_rate_service.go`

真正把这几件事拼起来的是 `backend/internal/service/gateway_service.go` 里的按量结算命令构造逻辑。它的关键判断可以概括成：

```go
if 是订阅计费 {
    cmd.SubscriptionCostUSD = p.Cost.TotalCost
} else {
    cmd.BalanceCostUSD = p.Cost.ActualCost
    cmd.BalanceCostCNY = p.balanceChargeAmountCNY()
    cmd.FXRateUSDCNY = p.ChargeSnapshot.FXRateUSDCNY
}
```

这段逻辑决定了一个非常重要的现象：

- 同样都是一次请求
- 如果它走订阅计费，就累计到订阅的 `USD` usage 窗口里
- 如果它走标准余额计费，就把 `USD` 成本换算成 `CNY` 去扣用户余额

所以“汇率为什么主要影响标准余额用户，而不是直接改写订阅权益”，答案就藏在这里。

这意味着：

- 当 `USD/CNY` 上升时，同样一笔 `USD` usage 会扣掉更多人民币
- 当 `USD/CNY` 下降时，同样一笔 `USD` usage 会扣掉更少人民币

### 一个最容易理解的例子

假设某次请求的真实 usage 成本是 `1.00 USD`。

如果当时系统采用的有效汇率是：

- `1 USD = 7.30 CNY`，那大致扣 `¥7.30` 再叠加安全边际
- `1 USD = 7.80 CNY`，那大致扣 `¥7.80` 再叠加安全边际

对用户来说，调用内容没变，消耗的 token 也没变，但人民币余额消耗速度变了。

这就是汇率波动对购买行为最直接的影响：

- 用户会更关心“现在充值值不值”
- 用户会更关心“同样的余额还能用多久”
- 当人民币变弱时，用户更可能转向订阅套餐或减少高成本模型调用

### 项目没有让汇率完全裸奔

为了避免实时汇率接口短暂异常或短时波动把结算打乱，项目用的是偏保守的汇率结算策略。

设计文档里明确写了这类思路：

- 在线汇率
- 最近一次成功汇率
- 底线汇率
- 安全加价

相关资料：

- `docs/plans/2026-04-15-dynamic-fx-billing-plan.md`
- `backend/internal/service/exchange_rate_service.go`

这带来的用户侧结果是：

- 人民币余额扣费不是随便拍脑袋算的
- 系统即使临时拿不到在线汇率，也不会直接失控
- 每次扣费都可以追溯当时用了什么汇率

### 系统是如何稳定、可靠获取汇率的

上面我们一直在说“在线汇率 + 最近成功汇率 + 底线汇率”，但如果不解释“系统到底怎么拿汇率”，这套设计还是会显得很抽象。

从源码看，Sub2API 为了让汇率获取尽量稳定，至少做了 6 层保护。

#### 1. 先有默认配置，再允许运行时覆盖

`defaultBillingFXSettingsFromConfig(...)` 会先给出一套默认值：

- 默认启用汇率服务
- 默认 provider 是 `default`
- 默认回退汇率 `FallbackRate = 7.2`
- 默认缓存 `CacheTTLSeconds = 600`
- 默认超时 `TimeoutMS = 3000`
- 默认安全边际 `SafetyMargin = 0.02`
- 默认在线汇率地址是 `https://open.er-api.com/v6/latest/USD`

这意味着系统不是“必须先手工配置好一切才工作”，而是开箱就有一套可运行的汇率策略。

同时，运行时设置又可以覆盖这些默认值，例如：

- `billing_fx_provider`
- `billing_fx_fallback_rate`
- `billing_fx_cache_ttl_seconds`
- `billing_fx_timeout_ms`
- `billing_fx_safety_margin`
- `billing_fx_last_success_rate`

这一步的好处是：

- 默认可用
- 又保留可运维性

#### 2. 先读进程内缓存，避免每次请求都联网

`ResolveUSDCNYRate(...)` 不会一上来就访问外部汇率接口。

它会先调用 `getCached()`；如果缓存还没过期，就直接返回缓存中的 `ResolvedExchangeRate`。

默认缓存时间是 600 秒，也就是 10 分钟。

所以它更像是：

- 先按时间片刷新汇率
- 再在这个时间片内复用结果

而不是：

- 每个 API 请求都去查一次实时汇率

这能明显降低：

- 外部接口压力
- 汇率接口抖动对结算链路的影响
- 每次 usage 结算的额外延迟

#### 3. 用 `singleflight` 合并并发拉取

即使缓存失效，系统也不会让大量并发请求同时冲出去查汇率。

`ResolveUSDCNYRate(...)` 里用了 `singleflight.Group`：

```go
value, err, _ := s.sf.Do("usd_cny", func() (any, error) {
    ...
})
```

这意味着：

- 缓存失效的那个瞬间
- 就算很多请求同时需要汇率
- 真正联网获取汇率的通常也只有一次
- 其他请求会复用这次结果

这一步很重要，因为它避免了缓存到期时对第三方汇率接口形成瞬时洪峰。

#### 4. 外部请求有明确超时，不会无限等待

联网获取汇率时，代码会用：

```go
context.WithTimeout(..., time.Duration(settings.TimeoutMS)*time.Millisecond)
```

包住请求。

默认超时是 3000ms，也就是 3 秒。

这表示：

- 第三方接口慢了，不会一直拖住本系统
- 汇率接口异常时，系统会尽快进入回退逻辑

对 API 网关来说，这是可靠性关键点，因为汇率获取属于高频结算链路的一部分，不能让一个外部接口把整条链路卡死。

#### 5. 在线取值失败时，会自动降级到安全值

如果发生下面这些情况：

- provider 不存在
- HTTP 请求失败
- 响应状态码异常
- JSON 解析失败
- 返回结果里没有有效 `CNY` 汇率
- 汇率值非正数

系统都不会直接报废，而是改走 `resolveFallbackExchangeRate(...)`。

而 `resolveFallbackExchangeRate(...)` 又不是简单返回某个固定硬编码数，它会继续比较：

- `LastSuccessRate`
- `FallbackRate`

如果最近成功汇率更高，就优先用最近成功汇率；否则用底线汇率。

这就是为什么它的失败处理仍然是“可解释、可复算”的，而不是异常时随便给个数字糊过去。

#### 6. 最近成功汇率会持久化，避免重启后完全失忆

一旦在线拉取成功，系统会调用：

- `UpdateBillingFXLastSuccess(...)`

把最近成功汇率和时间写回设置层。

这一步的价值很大：

- 如果只放在内存里，进程重启后最近成功值就丢了
- 如果是多实例部署，不同实例也可能拿不到同一份最近成功值

持久化之后，系统即使遇到：

- 服务重启
- 第三方汇率接口暂时故障

也仍然有机会回退到“最近一次已知成功值”，而不是完全从零开始。

### 所以，它的可靠性来自分层容错，而不是赌外部接口永远正常

把整个流程串起来看，实际是这样的：

1. 先读默认与运行时配置
2. 再读进程内缓存
3. 缓存失效时用 `singleflight` 合并并发请求
4. 带超时去访问外部 provider
5. 成功就更新缓存和最近成功值
6. 失败就退回到最近成功值或底线汇率

这就是为什么这套汇率获取机制可以说是“稳定、可靠”的：

- 它允许在线汇率波动
- 也允许在线接口失败
- 甚至允许服务进程重启
- 但仍然尽量保证每次结算都有一个明确、可解释、可复现的汇率来源

### 这也解释了一个常见现象

有时管理员会发现：

- 外部市场汇率已经变化了
- 但系统短时间内还没立刻跟着跳

这通常不是 bug，而是因为系统本来就用了：

- 10 分钟缓存
- 最近成功值
- 底线汇率
- 安全边际

它的设计目标从来不是“秒级追市场”，而是“在可接受的新鲜度内，优先保证结算稳定性和经营可控性”。

背后的具体代码原因也很清楚：

- `ResolveUSDCNYRate(...)` 先查设置，再查进程内缓存
- 如果缓存没有，再通过 `singleflight` 合并并发请求，避免同一时刻打爆外部汇率接口
- 实时拉取成功后，不会直接裸用，而是取 `live_rate`、`fallback_rate`、`last_success_rate` 中更保守的值
- 最终再叠加 `SafetyMargin`

等价地说，代码实际执行的不是“查到多少就用多少”，而更接近：

```text
effective_rate = max(live_rate, fallback_rate, last_success_rate) + safety_margin
```

这就是为什么用户有时会感觉：

- “今天汇率似乎没有按实时最低价来算”

这通常不是 bug，而是因为结算逻辑本来就偏保守。

另外，usage 账单不是只在内存里算完就结束。`backend/internal/repository/usage_billing_repo.go` 会把这些信息先写入 `usage_billing_dedup`，再执行余额扣减和 quota 增量：

```go
if cmd.BalanceCostCNY > 0 {
    deductUsageBillingBalance(..., cmd.BalanceCostCNY)
}

result.ChargeSnapshot = &UsageChargeSnapshot{
    ChargedAmountCNY: cmd.BalanceCostCNY,
    FXRateUSDCNY:     cmd.FXRateUSDCNY,
    FXRateSource:     cmd.FXRateSource,
}
```

这解释了另一个现象：

- 为什么重复回调、重试请求或网关补写，不应该把同一笔 usage 反复扣很多次

因为仓库层已经按 `request_id + api_key_id` 做了幂等 claim。

还有一个容易被忽视、但和用户体验有关的细节：

- `createBalanceSubscriptionOrderInTx(...)` 购买订阅时会要求 `Balance >= amount`
- `userRepository.DeductBalance(...)` 的按量扣费路径却允许余额被扣成负数

也就是说：

- 买套餐时，系统要求你先有足够人民币余额
- 但一条已经放行的 API 请求，为了不中断，会允许本次扣费把余额打到负数，后续再由中间件阻止继续请求

这也是为什么用户偶尔会看到“本次请求成功了，但余额已经负了”的源码原因。

## 订阅套餐用户会受到什么影响

订阅用户也会被汇率影响，但方式和按量余额用户不同。

### 套餐标价本身通常不随汇率实时跳动

当前项目约定很清楚：

- 订阅套餐价格按人民币展示和支付
- 订阅套餐的 usage 限额仍按美元语义展示
- 前端不做自动汇率换算

相关文档：

- `docs/PAYMENT_CN.md`
- `README.md`

相关前端展示：

- `frontend/src/components/payment/SubscriptionPlanCard.vue`
- `frontend/src/views/PublicPricingView.vue`

这意味着一个套餐常见会长这样：

- 价格：`¥99 / 月`
- 额度：`$15 / 月` 或某个 `daily_limit_usd` / `monthly_limit_usd`

这不是文案层碰巧这样写，而是支付和展示链路都在用同一套数据结构。

后端创建订阅订单时，会直接把：

```go
amount = plan.Price
```

写进订单；也就是说订阅订单金额源头就是套餐配置里的 `Price`，不是“先拿美元额度再乘当天汇率”。

对应代码在 `backend/internal/service/payment_order.go`。

而前端展示时又明确区分：

- 套餐价格走 `formatPaymentAmount(selectedPlan.price)`
- 套餐额度走 `formatUsageCost(selectedPlan.daily_limit_usd)`

因此，一个套餐为什么能长期表现成“固定人民币售价 + 固定美元额度”，答案不是抽象业务概念，而是这两条代码链路一起把它钉死了。

### 汇率波动会改变用户对“这个套餐划不划算”的主观判断

因为套餐价格是固定人民币，而套餐额度的业务语义是美元。

所以：

- 当 `USD/CNY` 上升时，同样一个人民币定价的套餐，看起来会更“划算”
- 当 `USD/CNY` 下降时，同样一个人民币定价的套餐，看起来会更“不占便宜”

注意，这里说的是“用户感知价值”会变，不是系统会自动改价。

系统当前没有做“汇率一变，套餐价格自动跟着变”的逻辑。

### 已购订阅的核心权益不会被汇率回头改写

这是项目里很重要的用户权益保护点。

用户一旦买下订阅，系统更关心的是这些快照和权益：

- 套餐价格快照
- 套餐有效期
- 当前计费周期起点
- 日/周/月额度窗口

相关代码：

- `backend/internal/service/subscription_service.go`
- `backend/internal/service/subscription_upgrade_service.go`

其中最关键的不是“订阅对象存在”，而是订阅对象里保存了套餐快照。

`applySubscriptionPlanSnapshot(...)` 会把这些值写进当前订阅：

- `CurrentPlanID`
- `CurrentPlanName`
- `CurrentPlanPriceCNY`
- `CurrentPlanValidityDays`
- `CurrentPlanValidityUnit`
- `BillingCycleStartedAt`

这一步非常重要，因为它解释了两个现象：

1. 为什么已购订阅不会因为套餐后来改价就自动回头改写
2. 为什么升级时系统能按“你当初买的那个价格”来算剩余价值

这表示：

- 已买到的订阅不会因为第二天汇率变了，就突然缩短有效期
- 已买到的订阅不会因为第二天汇率变了，就自动重算成另一套价格
- 已经生效的套餐窗口和 `*_usd` 限额，仍按订阅规则继续执行

换句话说，汇率更多影响的是“新用户要不要买”“老用户下个周期要不要续”，而不是随意改写已生效权益。

再往深一层看，订阅 usage 的累计方式也在强化这一点。

`SubscriptionService.RecordUsage(...)` 直接调用的是：

```go
IncrementUsage(ctx, subscriptionID, costUSD)
```

也就是订阅窗口累计的是 `USD` usage，而不是当时折算后的 `CNY`。

后面计算进度时，用的也是：

- `sub.DailyUsageUSD`
- `group.DailyLimitUSD`
- `sub.WeeklyUsageUSD`
- `group.WeeklyLimitUSD`
- `sub.MonthlyUsageUSD`
- `group.MonthlyLimitUSD`

所以对订阅用户来说，汇率波动不会把已经买到的套餐窗口重新换算成人民币额度；它只是改变用户心里对这个套餐“值不值”的判断。

## 补差价升级为什么基本不受实时汇率冲击

订阅升级走的是另一套规则。

当前升级逻辑会根据：

- 原套餐的人民币价格快照
- 当前周期剩余比例
- 目标套餐的人民币价格

来计算：

- `credit_cny`
- `payable_cny`

相关代码：

- `backend/internal/service/subscription_upgrade_service.go`

这意味着升级是按人民币快照和剩余价值来算差价，不是按“此刻美元汇率重新估值旧套餐”。

源码里的关键公式几乎是明写的：

```go
remainingRatio := expiresAt - now / cycleDuration
creditCNY := CurrentPlanPriceCNY * remainingRatio
payable := max(targetPlan.Price - creditCNY, 0)
```

这三行就是升级补差价的核心。

因此会出现几个非常具体的现象：

- 如果旧套餐剩余很多天，`creditCNY` 就高
- 如果旧套餐快到期，`creditCNY` 就低
- 如果旧套餐剩余价值已经覆盖目标套餐，升级订单金额可以直接变成 0

这些现象都不是汇率临时驱动的，而是 `CurrentPlanPriceCNY + BillingCycleStartedAt + ExpiresAt` 这组快照字段驱动的。

它对用户权益的意义是：

- 升级补差价可预期
- 前端展示的可抵扣金额更容易解释
- 不会因为用户升级时恰好碰上汇率波动，就把旧套餐剩余价值算得忽高忽低

这里还隐藏着另一个现象：为什么有些历史订阅明明还在有效期，却“暂不支持升级”。

原因不是运营故意限制，而是代码明确要求：

- `CurrentPlanID`
- `CurrentPlanPriceCNY`
- `CurrentPlanValidityDays`
- `BillingCycleStartedAt`
- `CurrentPlanValidityUnit`

这些快照字段必须齐全；缺任意一个，就直接返回 `UPGRADE_SOURCE_SNAPSHOT_MISSING`。

所以“历史无快照订阅不能补差价升级”并不是文档说说，而是升级服务在源码里强校验出来的结果。

## 从“购买行为”角度，汇率会诱导哪些用户选择

如果站点同时提供“余额充值按量计费”和“订阅套餐”两种购买方式，汇率会改变用户的选择倾向。

常见规律通常是：

- 人民币偏弱时，用户会更倾向于选择固定价的订阅套餐，因为按量余额的消耗会显得更快
- 人民币偏强时，一部分轻度用户会更愿意继续按量充值，因为单次调用折算成人民币更便宜
- 对高频用户来说，汇率越不稳定，越可能偏好价格更确定的订阅

也就是说，汇率不仅影响“扣多少钱”，还会影响：

- 用户充值还是订阅
- 用户是否升级到更高档套餐
- 用户是否减少高成本模型调用
- 用户是否更频繁地查看 usage 和余额页面

---

## 从“用户权益”角度，当前项目提供了哪些保护

### 保护一：余额和 usage 成本分币种建模

这样做虽然看起来麻烦，但能避免最危险的错误：

把美元 usage 数值直接当人民币扣款。

### 保护二：usage 日志保留人民币扣款和汇率快照

用户和管理员后续对账时，至少可以解释这些问题：

- 这次请求实际扣了多少人民币
- 当时采用的汇率是多少
- 这个汇率来自哪里

相关前端页面已经展示了这类信息：

- `frontend/src/views/user/UsageView.vue`

相关字段：

- `charged_amount_cny`
- `fx_rate_usd_cny`
- `fx_rate_source`

对应地，前端 usage 页面也不是只展示一个美元数字就结束。

它会：

- 先用 `formatUsageCost(row.actual_cost)` 展示美元成本
- 再在下面用 `formatUsageChargeAmount(row.charged_amount_cny)` 展示人民币扣款
- tooltip 里继续展示 `fx_rate_usd_cny` 和 `fx_rate_source`

所以用户看到“一笔 usage 同时有美元成本和人民币扣款”时，不要把它理解成重复收费；它其实是在展示同一笔 usage 的两种语义：

- 一种是上游成本语义
- 一种是钱包结算语义

### 保护三：订阅权益按订阅规则执行，不按实时汇率反复改价

这能防止用户刚买完套餐，系统就因为汇率变动反悔，属于用户可预期性的核心保障。

### 保护四：升级补差价依赖套餐快照，不依赖临场重算

这减少了升级过程中的解释成本，也降低了争议。

---

## 也要看到当前设计的边界

这套设计并不是“汇率影响彻底消失”，而是“把影响收束到可解释的地方”。

当前边界主要有这些：

- 标准余额按量计费用户，仍然会真实感受到汇率变化带来的余额购买力波动
- 套餐价格如果长期固定不调，汇率长期偏离后，用户会明显感知某些套餐过于划算或不划算
- 前端当前不是多币种实时展示系统，主要仍按人民币价格和美元额度并列展示

所以这套方案更像是：

- 对按量计费，接受汇率影响，但要求可追溯、可解释
- 对订阅权益，尽量冻结用户已买到的承诺，不让汇率回头改写

---

## 当前汇率策略更偏向谁

如果一定要问“目前这套汇率处理，整体更有利于用户还是更有利于管理员”，更准确的回答是：

- 在按量余额计费上，当前策略更偏向管理员
- 在已购订阅权益保护上，当前策略更偏向用户稳定预期
- 综合来看，系统整体略偏管理员一侧

原因不难从源码里看出来。

按量计费时，系统不是取“对用户最省钱”的最低实时汇率，而是走更保守的结算逻辑：

- 先比较 `live_rate`
- 再比较 `last_success_rate`
- 再比较 `fallback_rate`
- 最后再叠加 `safety_margin`

这等价于让最终结算汇率更不容易偏低。对平台来说，这能减少下面这些风险：

- 上游成本已经按美元真实发生，但用户人民币扣款过低
- 汇率接口短暂异常时把人民币扣费压得过低
- 短期波动把本来就很薄的利润空间直接吞掉

对应代码可以直接看：

- `backend/internal/service/exchange_rate_service.go`
- `backend/internal/repository/usage_billing_repo.go`

所以，如果只看按量余额用户，平台确实是在主动保护自己不被汇率波动“白白吃亏”。

### 但这不是为了压榨消费者

这里很重要的一点是：这种设计的出发点，不应理解为“开发者想利用汇率多收用户一点钱”，而更应该理解为一种运营稳定性设计。

更具体地说，它要解决的是长期经营中的三个现实问题：

- 如果平台长期按偏低汇率结算，表面上用户当下更便宜，但平台利润会被持续侵蚀
- 利润一旦被侵蚀到撑不住，后面通常不是“大家继续便宜使用”，而是平台被迫频繁涨价、缩容、限制模型、减少号池，甚至停止服务
- 对多数真实用户来说，最差体验往往不是“每笔扣费略保守”，而是“平台突然不稳定、套餐频繁改价、服务时有时无”

换句话说，这套策略更接近：

- 牺牲一部分“短期内对用户最便宜”的结算结果
- 换取平台利润的可预测性、供给的稳定性、价格体系的持续性

它追求的不是一次性多赚，而是“不要因为汇率噪声把平台经营打穿”。

从运营哲学上讲，这是一种偏长期主义的做法：

- 先守住平台不亏损或不轻易被波动拖垮
- 才有条件持续提供模型、维护订阅体系、稳定支付和售后

如果平台未来想进一步向用户友好侧再走一步，更合理的方向通常不是取消这套保守机制，而是：

- 在文档里更清楚地披露“按量结算会使用保守汇率 + 安全边际”
- 给用户更多订阅型、固定价型产品，让重度用户获得更稳定的价格预期
- 在运营利润更稳后，再审慎下调 `fallback_rate` 或 `safety_margin`

也就是说，这套机制的更合理理解应当是：

> 它首先是在保护平台长期可持续运营，其次才是在具体某一笔账单上决定平台和用户谁更占便宜。

## 最后总结

在 Sub2API 里，汇率波动并不是平均地影响所有用户。

- 对标准余额按量计费用户，影响是直接的：同样的美元 usage，会换算成不同的人民币扣款
- 对订阅套餐用户，影响是间接的：更多体现在“值不值得买”的主观感受上，而不是系统实时改写已购权益

项目当前的核心取向很明确：

- 让按量计费的汇率影响变得可审计、可解释
- 让订阅权益保持稳定、避免被实时汇率反复改写
- 让平台在汇率波动下仍能维持利润稳定与持续供给，而不是为了短期便宜把经营风险转成后续的涨价、缩容或服务不稳定
