# 管理员参数设置最佳实践

本文面向 Sub2API 管理员，记录分组配置中一些容易误解、但会直接影响用户接入方式与路由行为的参数解释与推荐设置方式。

## OpenAI Messages 调度配置

### 参数名称

- `允许 /v1/messages 调度`

### 这个参数本质上控制什么

这个开关只对 `OpenAI` 平台分组有意义。

开启后，这个 OpenAI 分组的 API Key 可以接收 `Claude /v1/messages` 风格的请求。系统会把 Anthropic Messages 请求转换成 OpenAI `Responses` 请求发送给上游，再把上游响应转换回 Anthropic Messages 风格返回给客户端。

换句话说，这不是一个普通的性能优化参数，而是一个“是否允许这个 OpenAI 分组作为 Claude 协议兼容入口使用”的开关。

### 开启后会发生什么

- 用户可以拿这个分组的 API Key 访问 `/v1/messages`
- `Claude Code`、Anthropic SDK、以及所有基于 Anthropic Messages 协议的客户端，都有机会直接接入这个分组
- 请求中的 Claude 模型名会根据映射规则，自动转发到对应的 OpenAI / Codex 模型

### 关闭后会发生什么

- 这个分组不接受 `/v1/messages` 调度
- 用户如果仍然向这个分组发起 `/v1/messages` 请求，会收到权限错误
- 该分组将只适合 OpenAI 自己的接口使用场景，例如 `/v1/responses`

## 什么情况下建议开启

建议开启的典型场景：

- 你希望这个 `OpenAI` 分组被 `Claude Code` 使用
- 你希望这个 `OpenAI` 分组兼容 Anthropic SDK 或第三方 Claude 客户端
- 你在做“Claude 协议兼容 OpenAI 模型”的分组，对外提供统一的 `/v1/messages` 接入体验
- 你希望用户继续使用 Claude 模型名，但底层实际消费的是 GPT / Codex 模型

一句话判断：

如果这组 Key 需要给 Claude 生态客户端使用，就开启。

## 什么情况下建议关闭

建议关闭的典型场景：

- 这组就是纯 OpenAI 用途，不希望兼容 Claude 协议
- 这组只打算提供 `/v1/responses` 等 OpenAI 风格接口
- 你不希望用户误把这组 Key 当作 Claude Key 来用
- 这组的账号池、计费策略或风控策略不适合承接 `/v1/messages` 流量
- 你希望把“OpenAI 直连分组”和“Claude 兼容分组”严格分开管理

一句话判断：

如果这组 Key 不打算给 Claude 生态客户端使用，就关闭。

## 默认映射模型是什么意思

当前默认映射规则为：

- `Opus -> gpt-5.4`
- `Sonnet -> gpt-5.3-codex`
- `Haiku -> gpt-5.4-mini`

含义是：

- 当请求命中 `claude-opus-*` 时，默认会使用 `gpt-5.4`
- 当请求命中 `claude-sonnet-*` 时，默认会使用 `gpt-5.3-codex`
- 当请求命中 `claude-haiku-*` 时，默认会使用 `gpt-5.4-mini`

如果管理员没有显式填写映射值，系统也会回落到上述默认值。

## 如何理解这三个系列的推荐映射

可以按“大致定位”理解，而不要把它当成严格的一一等价关系：

- `Opus` 通常对应更高能力档位，因此映射到主力高能力模型更合适
- `Sonnet` 通常对应通用主力模型，因此适合映射到兼顾能力与成本的主力模型
- `Haiku` 通常对应轻量、快速、低成本模型，因此适合映射到轻量模型

如果你的目标是让用户“感觉像在用 Claude”，而底层实际是 OpenAI 模型，那么系列映射应尽量保持这个能力梯度，而不是随意混配。

## 推荐配置策略

### 面向 Claude Code 的 OpenAI 分组

推荐：

- 开启 `允许 /v1/messages 调度`
- 明确配置 `Opus / Sonnet / Haiku` 的默认映射
- 如果某个具体 Claude 型号需要单独落到不同模型，再补充“精确模型映射”

适合：

- 面向 `Claude Code`
- 面向 Anthropic SDK
- 面向使用 Claude 协议的第三方客户端

### 面向 OpenAI 原生客户端的分组

推荐：

- 关闭 `允许 /v1/messages 调度`

适合：

- 只面向 OpenAI SDK
- 只提供 `/v1/responses`
- 不希望产生 Claude 协议兼容语义

### 需要减少误用的生产环境分组

推荐：

- 对“专门给 Claude 客户端用”的分组开启
- 对“普通 OpenAI 分组”关闭
- 通过分组命名把用途写清楚，例如“OpenAI 原生”“Claude 兼容 OpenAI”

这样可以减少用户混用协议带来的排障成本。

## 管理员的实际决策模板

管理员在配置时，可以直接问自己一个问题：

“这组 API Key 要不要给 Claude Code、Anthropic SDK 或 Claude 协议客户端使用？”

- 如果答案是“要”，就开启
- 如果答案是“不要”，就关闭

这是最稳定、最不容易出错的判断标准。

## 额外注意事项

- 该开关仅对 `OpenAI` 平台分组有意义，非 OpenAI 分组不应依赖此参数
- 该参数改变的是“入口协议兼容性”，不是账号质量、并发能力或计费倍率
- 如果开启后没有合理设置模型映射，用户体验可能会与预期 Claude 系列能力不一致
- 如果运营侧已经承诺“本分组可用于 Claude Code”，则必须开启该参数，否则用户会在 `/v1/messages` 请求中直接报错

## 账号过滤控制

这一组参数控制的不是“用户能否访问分组”，而是“分组内哪些上游账号有资格参与调度”。

它们更接近账号池筛选器，而不是协议兼容开关。

### 仅允许 OAuth 账号

#### 这个参数是什么意思

开启后，这个分组只允许使用 `OAuth` 类型账号。

普通 `API Key` 类型账号不能加入这个分组；如果管理员尝试把 API Key 类型账号绑定到这个分组，系统会直接拒绝。复制其它分组账号到当前分组时，也会自动过滤掉 API Key 类型账号。

#### 开启后会发生什么

- 分组中的可用账号将被限制为 OAuth 账号
- API Key 类型账号无法加入该分组
- 从其它分组复制账号时，API Key 类型账号会被自动排除

#### 关闭后会发生什么

- 这个分组可以同时包含 OAuth 账号和 API Key 类型账号
- 调度时不会因为账号类型而额外过滤

#### 什么情况下建议开启

- 你希望这个分组专门服务于 `ChatGPT`、`Codex`、`Claude Code`、`Gemini Code Assist` 等 OAuth 登录体系账号
- 你希望这个分组只承接 OAuth 账号独有能力，不想混入普通 API Key 账号
- 你希望运维上把“OAuth 账号池”和“API Key 账号池”明确隔离

#### 什么情况下建议关闭

- 你希望这个分组混用 OAuth 账号和 API Key 账号
- 你当前账号池里大量是 API Key 类型账号，开启后会明显缩小可用账号数量

一句话判断：

如果你要按“账号类型”把 OAuth 账号单独隔离出来，就开启；否则关闭。

### 仅允许隐私保护已设置的账号

#### 这个参数是什么意思

开启后，调度器只会选择“已经完成隐私保护设置”的账号。

如果账号未完成隐私设置，系统会在调度时跳过它，并记录相应异常信息。

当前实现中：

- 对 `OpenAI` 平台，要求账号的 `privacy_mode == training_off`
- 对 `Antigravity` 平台，要求账号的 `privacy_mode == privacy_set`
- 对其他平台，可理解为没有额外 privacy 限制

#### 开启后会发生什么

- 未完成隐私设置的账号不会参与调度
- 分组整体可用账号数可能下降
- 如果账号池本来就少，极端情况下可能出现“没有可用账号”

#### 关闭后会发生什么

- 调度时不会因为 privacy 状态而额外过滤账号
- 账号是否完成隐私保护设置，不影响其被选中

#### 什么情况下建议开启

- 你对隐私保护、数据隔离、合规性有明确要求
- 你对外承诺这个分组只使用“关闭训练/完成隐私保护”的账号
- 你在做企业、团队、敏感业务或其他需要更强隐私保证的分组

#### 什么情况下建议关闭

- 你的账号还没有统一完成隐私设置
- 当前更关注可用性，不希望因为 privacy 过滤导致可用账号显著减少
- 你还处在账号池整理阶段，准备等全部配置好后再统一开启

一句话判断：

如果你要按“隐私合规状态”筛账号，就开启；如果当前更看重账号池可用性，就先关闭。

## 账号过滤控制的推荐组合

### 稳定优先的普通生产分组

推荐：

- `仅允许 OAuth 账号` 视账号池结构决定
- `仅允许隐私保护已设置的账号` 先关闭，等账号治理完成后再开启

适合：

- 刚开始搭建账号池
- 当前优先保证可用账号数量

### 合规优先的企业或敏感业务分组

推荐：

- 如果本来就是 OAuth 体系账号池，开启 `仅允许 OAuth 账号`
- 开启 `仅允许隐私保护已设置的账号`

适合：

- 企业客户
- 团队内敏感项目
- 明确承诺隐私保护策略的场景

### 面向 Claude Code 或 Codex 的专用分组

推荐：

- 如果该分组本来就是 OAuth 账号池，开启 `仅允许 OAuth 账号`
- 如果你对数据训练隔离有要求，再开启 `仅允许隐私保护已设置的账号`

适合：

- 面向开发工具链
- 需要明确控制账号来源与安全边界

## 快速理解这两个参数

- `仅允许 OAuth 账号`：筛“账号类型”
- `仅允许隐私保护已设置的账号`：筛“账号状态/合规性”

管理员在设置时，可以先问自己两个问题：

“我是不是要把 OAuth 账号和 API Key 账号分开管理？”

- 如果是，开启 `仅允许 OAuth 账号`
- 如果不是，关闭

“我是不是要求这组流量只能落到已完成隐私保护设置的账号上？”

- 如果是，开启 `仅允许隐私保护已设置的账号`
- 如果不是，关闭
