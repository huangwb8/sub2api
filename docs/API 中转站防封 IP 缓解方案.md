# API 中转站防封 IP 缓解方案

## 背景

Sub2API 部署在单台服务器（如 RackNerd US VPS）上，所有用户的请求都经同一 IP 转发至上游（OpenAI / Anthropic / Google 等）。上游可能通过以下手段识别中转行为并封锁 IP：

- 单 IP 高并发 / 高请求量
- IP 属于数据中心段（非住宅 IP）
- 同一 IP 关联大量不同 API Key
- 请求模式异常（无自然随机性、无人类行为特征）
- TLS 指纹与正常客户端不符
- User-Agent / HTTP Header 特征异常

**核心问题**：Cloudflare CDN 仅保护入站流量（用户→你的服务器），出站流量（你的服务器→上游 API）直接暴露你的真实 IP。

**本文目标**：从易到难，按层级穷举所有可用的缓解手段。

---

## 检测原理：上游如何识别中转站

理解检测机制才能对症下药。上游通常不会只看单一信号，而是综合多种指标打分：

| 信号 | 说明 |
|------|------|
| IP 信誉 | 数据中心 IP 段天然低信誉；ASN 归属 VPS 商的 IP 更容易被标记 |
| 请求量密度 | 个人用户日均几十到几百次请求；中转站可能达到数千至数万次 |
| Key 关联数 | 正常用户一个 IP 对应 1-2 个 Key；中转站一个 IP 可能关联数十个 Key |
| 时间分布 | 人类用户有昼夜节律；中转站请求可能 24 小时均匀分布 |
| 并发模式 | 个人用户极少同时发起 5+ 并发请求；中转站经常高并发 |
| TLS 指纹 | 不同客户端（浏览器、Python SDK、Go HTTP）有不同 JA3/JA4 指纹 |
| HTTP Header | 异常的 Header 顺序、缺失的预期 Header、非标准 User-Agent |

## 第一层：Sub2API 已有功能（零成本，立即启用）

### 1.1 为每个上游账号配置独立代理

Sub2API 原生支持为每个上游账号绑定代理（HTTP / HTTPS / SOCKS5）。这是最直接有效的分散 IP 的方式。

**配置方式**：在管理后台为每个上游账号设置 Proxy URL。

**效果**：不同账号的请求从不同 IP 发出，打破"单 IP 关联多 Key"的检测信号。

### 1.2 启用 TLS 指纹伪装

Sub2API 内置 TLS 指纹伪装功能，可模拟 Claude CLI / Node.js 等正常客户端的 TLS 握手特征。

```yaml
gateway:
  tls_fingerprint:
    enabled: true
```

**效果**：避免因 Go 标准库的 TLS 指纹被识别为非浏览器/非官方 SDK 客户端。

### 1.3 连接池隔离模式

```yaml
gateway:
  connection_pool_isolation: "account_proxy"
```

`account_proxy` 模式确保每个「账号 + 代理」组合使用独立的连接池，避免不同账号的请求在 TCP 层面被关联。

### 1.4 合理设置并发限制

为每个上游账号设置合理的并发上限，避免单账号瞬间高并发触发上游限速：

```
Account Concurrency: 建议 3-5（视账号等级而定）
```

### 1.5 账号调度器的负载均衡

Sub2API 的账号调度器会根据实时错误率和延迟自动选择最优账号，自然实现了请求在多账号间的分散。确保你添加了足够多的上游账号。

## 第二层：代理基础设施（低成本，推荐优先实施）

### 2.1 购买住宅代理（Residential Proxy）

住宅代理的 IP 来自真实 ISP 用户，信誉远高于数据中心 IP。

**推荐类型**：
- 轮换住宅代理（Rotating Residential）— 每次请求自动换 IP
- 静态住宅代理（Static Residential）— 固定 IP 但来自住宅段

**配置示例**（以 SOCKS5 为例）：

```
socks5://user:pass@gate.proxyprovider.com:1080
```

在 Sub2API 管理后台将此代理绑定到对应的上游账号。

### 2.2 多 VPS + 自建代理

在多个不同提供商、不同地区购买低价 VPS，自建代理节点：

**推荐 VPS 商**（分散 ASN 风险）：
- RackNerd（美国）
- BandwagonHost（美国/加拿大）
- Vultr（多地区）
- Hetzner（欧洲）
- DigitalOcean（多地区）

**每台 VPS 上搭建代理**（任选一种）：

```bash
# 方案 A：SSH 隧道（最简单）
ssh -D 1080 -N -f user@vps-ip

# 方案 B：Dante SOCKS5 服务器
# 安装 dante，配置 socks5 监听

# 方案 C：Gost 代理（推荐，轻量灵活）
docker run --rm -p 1080:1080 ginuerzh/gost \
  -L "socks5://user:password@:1080"
```

**然后在 Sub2API 中为不同账号分配不同 VPS 的代理。**

### 2.3 ISP 代理（ISP Proxy）

ISP 代理兼具数据中心的速度和住宅 IP 的信誉：IP 注册在 ISP 名下，但托管在数据中心。

适合需要固定 IP + 高速 + 高信誉的场景，但价格较贵。

### 2.4 IPv6 代理池

许多上游服务支持 IPv6 访问。IPv6 地址空间极大（/64 前缀下有 2^64 个地址），可以从同一服务器获得大量不同出口 IP。

部分代理提供商支持 IPv6 轮换，每个请求分配不同的 IPv6 地址。

## 第三层：请求模式优化（中等成本，减少检测信号）

### 3.1 请求速率平滑

避免请求呈突发脉冲式。可通过以下方式平滑：

- **控制用户并发**：在订阅组级别设置合理的 RPM（每分钟请求数）上限
- **排队机制**：高峰期请求排队而非瞬间全部发出
- **业务量错峰**：如果用户集中在某时区，考虑引导分散使用时间

### 3.2 请求间隔随机化

人为的请求具有随机性。如果你的中转站请求间隔完全均匀（如精确每秒 1 次），这本身就是一个异常信号。

目前 Sub2API 的请求完全由用户行为驱动，天然具有随机性，不需要额外处理。但如果你通过脚本批量调用，务必加入随机抖动（jitter）：

```python
import random, time
time.sleep(random.uniform(0.5, 2.0))
```

### 3.3 合理的 HTTP Header

确保出站请求的 Header 尽量模拟正常客户端。Sub2API 在转发时会设置合适的 Header（如 `User-Agent`），但需注意：

- 不要在 Header 中泄露中转站信息
- 避免添加非标准的 `X-Forwarded-*` Header 被传递到上游
- 如果使用自定义 Header，确保与目标 SDK 的行为一致

### 3.4 流式响应保持自然间隔

如果上游检测流式响应的消费模式，过于规律的 chunk 消费时间也可能是异常信号。Sub2API 的流式转发是实时 relay，本身行为与正常客户端一致。

## 第四层：多区域架构（较高成本，适合规模化运营）

### 4.1 多节点部署 + 全局负载均衡

在不同地区部署多套 Sub2API 实例，通过 DNS 负载均衡将用户分散到不同节点：

```
用户 → Cloudflare DNS (负载均衡) → Sub2api 节点 A (美国)
                                → Sub2api 节点 B (欧洲)
                                → Sub2api 节点 C (亚太)
```

每个节点使用不同的出站 IP 访问上游，大幅降低单 IP 的请求密度。

### 4.2 Cloudflare Workers 边缘代理

利用 Cloudflare Workers 在边缘节点代理出站请求。Cloudflare 的 IP 信誉通常较高：

```
用户 → Cloudflare CDN → 你的服务器 → Cloudflare Worker → 上游 API
```

**注意**：Cloudflare Workers 有出站请求限制（子请求数量、执行时间），可能无法处理长时间流式请求。

### 4.3 Serverless 函数分发

使用 AWS Lambda / GCP Cloud Functions / Azure Functions 等无服务器函数作为出站代理：

- 每次调用使用不同的临时 IP
- IP 来自云厂商的信誉池，信誉通常高于小型 VPS
- 按调用次数计费，流量低时成本可控
- 可按区域部署函数，进一步分散

### 4.4 反向架构：请求从边缘发出

将传统架构「用户 → 中心服务器 → 上游」改为「用户 → 中心服务器（调度）→ 边缘节点 → 上游」：

```
用户 → Sub2API 主节点（仅做调度和计费）
              ↓
        ┌─────┼─────┐
        ↓     ↓     ↓
      边缘1  边缘2  边缘3  （实际发起上游请求）
        ↓     ↓     ↓
      上游 API
```

主节点只负责认证、计费和调度，实际的上游请求由分布式的边缘节点发出。这是最彻底的分散方案，但架构复杂度最高。

## 第五层：账号与 Key 管理策略（运营层面）

### 5.1 分散 Key 到不同 IP

核心原则：**一个 IP 关联的 API Key 数量越少越好**。

- 每 3-5 个上游账号共享一个代理出口
- 不要把所有账号都挂在同一个代理后面
- 定期轮换代理和账号的绑定关系

### 5.2 账号分级与隔离

将上游账号按用途分组：

| 组 | 用途 | 代理策略 |
|----|------|----------|
| 高优组 | 付费用户 | 独立住宅代理，严格控制并发 |
| 普通组 | 免费试用 | 共享代理，允许较高并发 |
| 备用组 | 故障转移 | 待机状态，仅在其他组不可用时启用 |

### 5.3 Key 轮换周期

定期更换 API Key（如果上游允许），避免长期关联。不过这个操作成本较高，通常作为最后手段。

### 5.4 账号来源多样化

避免所有上游账号来自同一个上游组织/项目。使用不同的：

- OpenAI Organization
- Anthropic Workspace
- 不同的计费账号

## 第六层：监控与应急响应（运营保障）

### 6.1 IP 健康监控

定期检查你的出站 IP 是否被上游标记：

```bash
# 检查 IP 信誉分数（免费工具）
curl https://ipinfo.io/{your-ip}
curl https://www.abuseipdb.com/check/{your-ip}

# 检查 IP 类型（是否被识别为数据中心）
# 使用 https://ipqualityscore.com/ 等服务
```

### 6.2 错误率告警

Sub2API 的账号调度器已内置错误率追踪（EWMA 算法）。当某 IP 的错误率突然升高时（可能是被封的前兆）：

- 监控 HTTP 429（Too Many Requests）的频率
- 监控 HTTP 403（Forbidden）的出现
- 关注连接超时的突然增加

建议设置告警：当某代理出口的 5 分钟错误率超过阈值时，自动通知。

### 6.3 快速切换预案

当某个 IP 被封时，需要能快速切换：

1. **预配置备用代理**：在 Sub2API 中提前配置好备用代理
2. **批量替换代理**：通过管理 API 批量修改账号的代理设置
3. **DNS 快速切换**：如果使用域名指向代理，修改 DNS 即可切换

### 6.4 日志审计

保留出站请求日志（至少保留 7 天），记录：
- 使用的代理 IP
- 请求时间
- 上游响应状态码
- 错误信息

这些数据有助于分析被封原因和规律。

## 第七层：合规与长期策略（最根本的解决方案）

### 7.1 审查上游 ToS

最根本的缓解手段是确保你的使用方式符合上游的服务条款：

- OpenAI 的使用政策是否允许 API 转发/再分发
- Anthropic 的服务条款对中间代理的态度
- Google 的 API 使用限制

如果上游明确禁止，所有技术手段只能延缓而非阻止封禁。

### 7.2 申请官方合作伙伴/经销商资格

部分上游提供官方的合作伙伴或经销商计划：

- OpenAI 有 API 合作伙伴计划
- 通过官方渠道获得的额度，转发行为可能被默许甚至鼓励

### 7.3 选择对中转友好的上游

如果某些上游对 API 转发持开放态度，优先使用这些上游的服务，降低合规风险。

### 7.4 透明化运营

如果你的服务面向终端用户，在服务条款中明确说明：
- 你是 API 的中间服务商，不是上游本身
- 用户的请求会经过你的服务器转发
- 这既是合规要求，也能降低用户期望落差

## 典型配置示例

以下是一个中等规模部署的推荐配置：

### 基础架构

```
用户 → Cloudflare CDN → Sub2API (RackNerd US)
                            ↓
                    ┌───────┼───────┐
                    ↓       ↓       ↓
              代理节点A  代理节点B  代理节点C
              (住宅IP)  (VPS-EU)  (VPS-AP)
                    ↓       ↓       ↓
                  上游 API
```

### Sub2API 配置

```yaml
gateway:
  connection_pool_isolation: "account_proxy"
  max_idle_conns: 2560
  max_idle_conns_per_host: 120
  max_conns_per_host: 1024

  tls_fingerprint:
    enabled: true
```

### 账号代理分配

| 账号 | 代理 | 并发上限 |
|------|------|----------|
| Account-01 | socks5://user:pass@proxy-a:1080 | 3 |
| Account-02 | socks5://user:pass@proxy-a:1080 | 3 |
| Account-03 | socks5://user:pass@proxy-b:1080 | 5 |
| Account-04 | socks5://user:pass@proxy-b:1080 | 5 |
| Account-05 | socks5://user:pass@proxy-c:1080 | 3 |
| Account-06 | socks5://user:pass@proxy-c:1080 | 3 |

每个代理出口承载 2 个账号，并发合理分散。

## 常见误区

### 误区 1：「Cloudflare CDN 能保护我的出站 IP」

**错误**。Cloudflare 仅代理入站流量。你的服务器直接与上游 API 建立 TCP 连接，出站 IP 对上游完全可见。

### 误区 2：「HTTPS 加密就能防止检测」

**错误**。HTTPS 加密的是请求内容，但 IP 地址、TLS 握手特征、请求频率等元数据仍然可见。上游不需要知道你请求了什么，仅凭元数据就能判断是否为中转。

### 误区 3：「换一个 VPS 商就安全了」

**不够**。如果请求模式不变（高并发、均匀分布、大量 Key 关联），换 IP 只是时间问题。必须同时优化请求模式和分散 IP。

### 误区 4：「上游不会封付费用户的 IP」

**天真**。上游的反滥用系统通常是自动化的，不会区分付费与否。大量异常请求从同一 IP 发出，系统会自动标记。

### 误区 5：「代理会拖慢所有请求」

**视情况而定**。优质住宅代理的延迟通常只增加 20-50ms。对于 LLM API 这种响应时间动辄数秒的服务，额外延迟几乎可以忽略。选择代理时应关注稳定性而非极致速度。

---

## 总结

防封是一个持续对抗的过程，没有一劳永逸的方案。核心思路是：

1. **分散出口 IP** — 代理/多节点
2. **模拟正常用户行为** — TLS 指纹、请求模式、并发控制
3. **减少关联信号** — 连接池隔离、账号代理绑定
4. **监控和快速响应** — IP 健康检查、错误率告警
5. **合规运营** — 了解上游政策，选择对中转友好的上游

Sub2API 已经在第 2、3、4 点提供了良好的基础设施支持（TLS 伪装、连接池隔离、账号调度）。最关键的第 1 点（分散出口 IP）需要你自行搭建代理基础设施。
