## 发现的问题

### P1: 502 频发导致账号反复 temp-unschedule（设计缺陷）

**现象**: 今日 23:00–23:18 期间，账号 1、2、4、15、16、18 遭遇密集 502 错误，全部被 temp-unschedule 1 分钟。从系统日志可以看到同一批账号在短时间内被反复 temp-unschedule：

```
23:00:56 account=16 → temp-unschedule 1m (502)
23:07:32 account=15 → temp-unschedule 1m (502)
23:08:21 account=4  → temp-unschedule 1m (502)
23:10:58 account=15 → temp-unschedule 1m (502)   # 15号 3分钟内第二次
23:17:09 account=2  → temp-unschedule 1m (502)
23:17:53 account=15 → temp-unschedule 1m (502)   # 15号 第三次
23:17:56 account=18 → temp-unschedule 1m (502)
23:17:58 account=1  → temp-unschedule 1m (502)
```

**根因**: temp-unschedule 机制采用固定时长（1 分钟），没有指数退避。当上游服务持续不可用时，同一账号会陷入"失败→冷却1分钟→立即再失败"的死循环。

**设计缺陷**:
- `scheduling_mechanism_settings.go` 中 `DurationMinutes` 为固定值
- 没有记录连续 temp-unschedule 次数
- 没有基于失败频率的升级机制
- 当多个账号同时反复 502 时，系统没有识别"全局性问题"并采取不同策略

**修复建议**:
1. 引入**指数退避**: 首次 1 分钟，二次 2 分钟，三次 4 分钟...上限 30 分钟
2. 添加**连续失败计数器**: 在 account extra 中记录 `temp_unschedule_consecutive_count`
3. 添加**全局异常检测**: 当同组内 >50% 账号同时 temp-unschedule 时，标记为"组级异常"而非"账号级异常"，避免逐个冷却浪费时间
4. 引入**账号健康评分**: 综合 temp-unschedule 频率、错误率、平均延迟，对不健康账号降权

### P2: 限速账号缺少预测性规避（设计缺陷）

**现象**: 账号 11、12、13 全部因 `codex_7d_used_percent` 达到 100% 而被限速，恢复时间长达数天：
- 账号 11: 重置于 5月1日（明天）
- 账号 12: 重置于 5月5日（5天后）
- 账号 13: 重置于 5月5日（5天后）

这意味着 GPT_Standard 组 12 个账号中有 3 个（25%）长期不可用，且都是在达到 100% 后才触发限速。

**设计缺陷**:
- `ratelimit_service.go` 仅在 `used_percent >= 100` 时才标记限速
- 调度器 `selectByLoadBalance()` 的评分因子中没有考虑限速接近度
- 没有在 80–90% 阈值时主动降权或减少分配

**修复建议**:
1. 引入**分级限速预警**:
   - 70%: 低优先级（减少调度权重）
   - 85%: 中优先级（仅处理高优先级请求）
   - 95%: 高优先级（仅处理紧急请求）
   - 100%: 完全停止
2. 在调度评分中加入**限速接近度因子**: `rateLimitFactor = 1 - (used_percent / 100)`
3. 优先将请求分配给限速余量最大的账号，而非仅看当前负载

### P3: 永久失效账号未自动禁用（设计缺陷）

**现象**: 账号 7 (`kxsw1-team-20260416`) 在 4月26日因 "Workspace deactivated (402)" 进入 error 状态，但至今 `schedulable=true`。该账号从 4月26日 16:24 起已完全无法使用，但系统没有自动将其从调度池中排除。

**设计缺陷**:
- `ratelimit_service.go` 中 `shouldDisable` 标志仅设置 `status=error`，未同时设 `schedulable=false`
- 402 workspace deactivated 是明确的永久性错误，但系统没有区分临时错误和永久错误
- 没有自动化的错误账号清理流程

**修复建议**:
1. 对明确的永久错误（402 deactivated、403 banned、credential expired）**自动设 `schedulable=false`**
2. 在 account status 变为 error 时，发送通知给管理员
3. 添加**账号健康巡检**: 定期检测 error 状态账号，超过 N 天未恢复则建议删除
4. 区分**临时错误**（网络超时、502）和**永久错误**（402、403、credential 失效），分别处理

### P4: 调度器不考虑限速接近度（设计缺陷）

**现象**: 从并发数据看，GPT_Standard 组有 12 个账号但当前仅 2 个在用（账号 4 和 15），其余要么限速要么 temp-unschedule。而 GPT_Premium 组有 5 个账号但当前 0 个在用。两个组之间的负载分布明显不均。

**设计缺陷**:
- 调度器 `selectByLoadBalance()` 的评分公式为:
  ```
  score = priorityFactor*0.3 + loadFactor*0.3 + queueFactor*0.2 + errorFactor*0.2
  ```
  缺少 `rateLimitProximityFactor`
- 账号在被选中前不会检查其限速余量
- 没有**跨组溢出**机制：当 Standard 组账号不足时，不会自动借用 Premium 组的空闲账号

**修复建议**:
1. 在调度评分中加入第5个因子: `rateLimitFactor = (100 - used_7d_percent) / 100`，权重 0.15
2. 重新分配现有因子权重，例如 `priority:0.25, load:0.25, queue:0.15, error:0.20, rateLimit:0.15`
3. 考虑实现**组间溢出**: 当主组可用账号 < 阈值时，向 fallback_group 借用（需配合配置）
4. 添加**账号利用率均衡**: 避免少数账号承担大部分流量而快速耗尽配额

### P5: 代理故障转移缺少健康评分（设计缺陷）

**现象**: 账号 7 和 12 都经历了 proxy_failover（代理故障迁移），从原始代理迁移到新代理。但迁移逻辑仅基于特定 HTTP 状态码（500/502/503/504 等），不考虑延迟退化或丢包率。

**设计缺陷**:
- `proxy_failover_service.go` 仅在 `RecordUpstreamFailure` 时触发评估
- 没有代理的**综合健康评分**（成功率 + 延迟 + 稳定性）
- 没有防止代理间反复跳转的机制（可能导致 A→B→A→B 循环）
- 没有定期探测所有代理的健康状态，只在出错时才被动触发

**修复建议**:
1. 引入**代理健康评分**: `health = successRate*0.4 + latencyScore*0.3 + stabilityScore*0.3`
2. 添加**主动探测**: 定期对所有代理执行轻量级连通性检测，而非仅在出错时被动迁移
3. 防止**迁移环路**: 记录近期迁移历史，禁止在短时间内（如 30 分钟）在相同两个代理间反复迁移
4. 考虑**延迟预警**: 当代理平均延迟超过阈值时，降权而非直接迁移

## 非设计缺陷问题

以下问题不属于 sub2api 设计缺陷，而是外部因素：

| 问题 | 性质 | 说明 |
|------|------|------|
| 上游 502 错误 | 外部 | OpenAI 服务侧临时不可用，sub2api 已有 temp-unschedule 机制应对 |
| 账号 7 workspace deactivated | 外部 | 上游 workspace 被停用，但 sub2api 应更好地自动处理 |
| Codex 7d 限速 | 外部 | OpenAI 的使用量限制，但 sub2api 应更好地规避 |

| 优先级 | 问题 | 预计工作量 | 影响范围 |

| P0 | P3: 永久错误账号自动禁用 | 1天 | 避免调度器每次都尝试不可用账号 |
