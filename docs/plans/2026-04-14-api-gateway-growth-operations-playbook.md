# Sub2API 从 10 人到 10 万人的运营扩容教程

**Goal:** 给正在运营 Sub2API API 站点的团队一份可执行的扩容手册，随着用户规模从 10 人增长到 10 万人，分阶段回答部署方式、服务器参数、安全措施、专人维护与关键架构决策，目标是稳定、高效、低风险地扩容，而不是盲目堆技术。

**Architecture:** 这份教程以当前仓库已有能力为基线来规划扩容路线：前期优先复用 `deploy/docker-compose.local.yml`、`deploy/Caddyfile`、`deploy/config.example.yaml` 和现有 Go 单体架构；中期优先做无状态横向扩容、托管数据库/Redis、异步任务外移、监控补齐；后期再按控制面/数据面拆分，并在 1 万到 10 万用户区间谨慎引入 Kubernetes、多可用区和平台化治理。

**Tech Stack:** Go、Gin、Ent、PostgreSQL、Redis、Docker Compose、Caddy/Nginx、云负载均衡、Prometheus/Grafana、WAF/CDN

## 先说结论

- `10 -> 1000` 用户阶段，不要急着上 Kubernetes；先把备份、监控、回滚、安全基线补齐。
- 这个项目最先容易出问题的通常不是 Go 网关本身，而是 `usage_logs` 增长、支付回调稳定性、Redis/数据库连接池、以及没有专人盯告警。
- 1 万用户以前，模块化单体通常仍然是最优解；5 万用户左右再优先拆“控制面 / 数据面”，而不是一开始拆十几个微服务。
- 真正需要专人维护的拐点，不是某个绝对用户数，而是“开始有稳定付费客户 + 夜间告警 + 对账/退款压力 + 升级回滚风险”同时出现的时候。

## 适用前提

- 假设你运营的是一个真实收费 API 站，而不是纯内部测试站。
- 假设当前主栈仍然是本仓库的 Go 单体 + PostgreSQL + Redis，而不是完全重写。
- 假设主要压力来自 API 转发、认证、计费、支付、流式响应和后台管理，而不是静态页面访问。
- 假设你的目标是“少折腾、可恢复、能赚钱”，不是为了技术简历而过度平台化。

## 仓库内基线能力

- 当前仓库已经提供 Docker Compose 部署基线，推荐从 `deploy/docker-compose.local.yml` 起步。
- 当前仓库已经提供 Caddy 反向代理示例，包含 TLS、`/health` 健康检查、真实 IP 透传、基础连接池和日志配置，可参考 `deploy/Caddyfile`。
- 当前仓库已经支持 PostgreSQL、Redis、JWT、TOTP、URL allowlist、CSP、响应头过滤、Turnstile、计费熔断、认证限流等安全与稳定性配置，可参考 `deploy/config.example.yaml` 与 `README.md`。
- 当前仓库已有使用量异步写入池、用户消息队列等能力，说明中期扩容时应优先做“异步链路治理”和“使用量数据治理”，而不是先推翻重写。

## 分阶段总览

| 阶段 | 用户规模 | 推荐部署形态 | 机器建议 | 运维投入 |
|------|----------|--------------|----------|----------|
| 起步期 | 10-100 | 单机或单主机 Docker Compose | 4C / 8G / 100G NVMe | 创始人兼维护 |
| 小规模商业化 | 100-1000 | 多副本应用 + 托管数据库优先 | 应用 2-3 台 4C-8C / 8G-16G | 兼职运维或轮值 |
| 成长期 | 1000-10000 | 无状态横向扩容 + 外部队列 + 统一监控 | 应用 2-4 台 8C / 16G | 1 名明确负责人 |
| 平台化前夜 | 10000-50000 | 控制面/数据面分离 + HA 数据层 | 应用 3-6 台 8C-16C / 16G-32G | 1 名专职运维/SRE |
| 平台化阶段 | 50000-100000 | 多可用区 + K8s 或等价平台化 | 多副本集群 + 托管 HA | 1-2 名专职 SRE |

## 起步期：10-100 用户

### 部署方式

- 使用 `deploy/docker-compose.local.yml`，不要默认暴露 PostgreSQL 和 Redis 到公网。
- 1 台主机即可承载应用、PostgreSQL、Redis，同机部署的优点是简单、可迁移、回滚快。
- 前面放 Caddy 或 Nginx，优先复用仓库已有反代模板，不要手写一套陌生配置。

### 服务器硬件参数

- 推荐 `4C / 8G / 100G NVMe` 起步。
- 如果同时跑管理后台、PostgreSQL、Redis 和定时任务，内存不要低于 `8G`。
- 磁盘优先 NVMe，不建议便宜但高延迟的系统盘，因为账单、日志、数据库都依赖磁盘稳定性。

### 数据库与 Redis

- PostgreSQL 单实例即可，但要开启每日备份。
- Redis 主要承担认证限流、短期缓存、会话类数据，开启 AOF 持久化。
- 先检查核心表索引是否齐全，尤其是 `usage_logs`、订单、API Key、订阅相关表。

### 安全措施

- 固定设置 `JWT_SECRET` 和 `TOTP_ENCRYPTION_KEY`，不能让重启后会话和 2FA 全部失效。
- 只开放 `80/443` 和必要的 SSH 管理端口，SSH 仅允许固定 IP。
- 打开 HTTPS，优先放在 Cloudflare 或等价 CDN/WAF 后面。
- 保持数据库和 Redis 仅内网访问。
- 配置 `server.trusted_proxies`、CORS 白名单、CSP、URL allowlist，不要把默认开发心态带到生产。

### 人员配置

- 不需要专职运维，创始人或主程兼维护即可。
- 但必须把部署、重启、备份、回滚、查日志写成脚本或文档，避免“只有脑子里知道”。

### 其它关键动作

- 至少建立 5 项监控：CPU、内存、磁盘、PostgreSQL 健康、Redis 健康。
- 每周做 1 次恢复演练：从备份恢复数据库到临时环境。
- 不要过早拆服务，不要为了“以后可能有 10 万用户”提前上复杂中间件。

### 进入下一阶段的触发条件

- 单机在高峰期 CPU 持续高于 60%。
- 夜间开始出现偶发告警或支付 webhook 延迟。
- 已经出现真实付费用户，且停机影响收入。

## 小规模商业化：100-1000 用户

### 部署方式

- 应用层扩成 `2-3` 个无状态副本，前面接云负载均衡、Caddy 或 Nginx。
- 数据库优先迁到托管 PostgreSQL；Redis 仍可自托管，但要保留持久化和密码保护。
- 继续保持单体仓库和单体部署逻辑，不要因为多副本就强行拆微服务。

### 服务器硬件参数

- 每台应用建议 `4C-8C / 8G-16G`。
- 托管 PostgreSQL 建议从 `4C / 8G` 或同等级规格起步。
- Redis 建议 `2C-4C / 4G-8G`，重点是低延迟和稳定，不是极限吞吐。

### 数据库与 Redis

- PostgreSQL 开始关注连接池、慢查询、自动备份和只读副本能力。
- Redis 至少启用密码、持久化、连接数监控。
- 后台读多写少的配置和查询结果可以开始做短 TTL 缓存。

### 安全措施

- WAF/CDN 做第一层防护，服务端限流和响应读取上限做第二层防护。
- 对认证、注册、登录、2FA、支付回调和管理后台接口建立单独限流策略。
- 检查反向代理是否正确传递流式连接、真实 IP、超时设置和大包体限制。
- 管理后台建议开启 2FA，运维入口不允许只靠弱密码。

### 人员配置

- 建议至少 2 名后端或 1 名后端 + 1 名全栈共同值守。
- 运维仍可兼职，但必须明确“谁收告警、谁能回滚、谁能恢复备份”。

### 其它关键动作

- 接入 Prometheus + Grafana，至少覆盖 5xx、P95 延迟、数据库连接数、Redis 命中率、支付 webhook 失败率。
- 发布流程从“直接改线上”升级为“构建镜像 -> 小流量验证 -> 全量发布 -> 可回滚”。
- 记录错误预算和 SLO，哪怕先只写最朴素的版本。

### 进入下一阶段的触发条件

- 需要定期处理对账、退款、补单。
- 使用量写入、通知、账单聚合已经挤占主进程资源。
- 单纯加应用副本已经不能明显改善高峰稳定性。

## 成长期：1000-10000 用户

### 部署方式

- 保持模块化单体，但把部署层按职责拆开：网关热路径、后台管理、支付回调、异步 worker 分开跑。
- 应用层维持 `2-4` 台实例，前面使用稳定的负载均衡。
- 开始引入外部队列或事件流，不再让所有异步任务都依赖进程内池。

### 服务器硬件参数

- 网关应用建议 `8C / 16G` 起步。
- PostgreSQL 建议 `8C / 16G-32G` 级别，并优先使用托管高可用版本。
- Redis 建议独立实例，不与应用同机混部。

### 数据库与 Redis

- `usage_logs` 是最需要提前治理的热点，建议在这个阶段开始做按时间分区、归档和聚合表。
- Redis 按用途拆分，至少把限流、会话、短期缓存区分开，避免互相影响。
- 对账和运营报表不要直接扫在线明细表，先做日级聚合。

### 安全措施

- 引入镜像扫描、依赖漏洞扫描和最小权限配置。
- Secrets 不再散落在手工 `.env` 中，建议迁移到云密钥管理或统一 Secret 管理方案。
- 重点保护支付、账务、用户管理和上游账号调度逻辑，建立审计日志。
- 上游账号保护限流必须落地，避免单个客户或单个模型把上游账号池打爆。

### 人员配置

- 建议 `3-4` 名后端，其中至少 `1` 人偏基础设施/性能。
- 这时通常已经需要明确的“平台负责人”，哪怕 title 还不是 SRE。

### 其它关键动作

- 建立日志聚合与链路追踪，至少打通“入口请求 -> 上游调用 -> 使用量落库 -> 账单聚合”。
- 建立异步任务重试、死信、补偿和幂等策略。
- 对 usage、支付、通知、清理任务做隔离，不让单个慢任务拖垮主链路。

### 进入下一阶段的触发条件

- `usage_logs` 清理或查询已经明显拖慢主库。
- 夜间告警频率上升，需要正式轮值。
- 不同业务流量互相影响，后台操作会伤到网关热路径。

## 平台化前夜：10000-50000 用户

### 部署方式

- 建议拆成“控制面 / 数据面”两条部署链路。
- 数据面只做鉴权、配额检查、上游选路、转发、热路径缓存。
- 控制面负责用户、订阅、支付、后台配置、报表和运营操作。
- 不一定拆仓，但必须资源隔离、发布隔离、告警隔离。

### 服务器硬件参数

- 数据面应用建议 `3-6` 台 `8C-16C / 16G-32G`。
- PostgreSQL 建议主备或托管高可用版，规格至少 `8C-16C / 32G-64G`。
- Redis 建议高可用部署，关注热 key 和连接数上限。

### 数据库与 Redis

- `usage_logs`、账单明细、审计日志必须明确冷热分层策略。
- 报表类需求使用聚合表、物化视图或分析库，不直接冲击主交易库。
- 读写分离和连接池治理应在这个阶段稳定运行。

### 安全措施

- 接入 DDoS 防护、运维审计、RBAC、镜像签名或可信制品链。
- 生产变更要求可审计，至少知道谁在什么时间发布了什么版本。
- 对高风险模型、高成本模型和大客户流量做独立限额和熔断策略。
- 核心配置变更建立双人复核机制，尤其是支付、配额、回调 URL、上游凭据。

### 人员配置

- 建议 `5-7` 名后端/平台相关成员。
- 至少需要 `1` 名专职运维/SRE，或 `0.5` 运维 + `0.5` 后端长期轮值。
- 开始建立正式 on-call、事故复盘、变更窗口和升级冻结规则。

### 其它关键动作

- 业务监控要超过系统监控：按模型毛利、上游成功率、租户异常峰值、账务延迟做看板。
- 压测和容量规划成为常规工作，而不是线上出问题才补做。
- 退款、补单、对账、坏账处理要有后台工具，而不是只靠 SQL 手工修。

### 进入下一阶段的触发条件

- 单可用区故障已经不可接受。
- 大客户 SLA、合规或跨地区访问开始成为商机要求。
- 发布、值班、容量和成本问题已经超出“一个人记得住”的范围。

## 平台化阶段：50000-100000 用户

### 部署方式

- 这时再考虑 Kubernetes、多可用区或等价的平台化方案。
- 数据面按地区或流量池独立扩容，控制面继续单独部署。
- 微服务不是必须；只有当支付、计费、网关调度已经需要独立演进时，再拆成少量边界清晰的服务。

### 服务器硬件参数

- 不再依赖单机规格，重点转向副本数、自动扩缩、跨可用区冗余和资源配额。
- 数据面维持多副本，预留高峰冗余，不要把平均流量当容量上限。
- 数据库和 Redis 尽量托管高可用，减少自建集群维护负担。

### 数据库与 Redis

- PostgreSQL 需要冷热分层：近 `30-90` 天在线，历史数据归档到分析库或对象存储。
- Redis 集群化时严格控制大 key、热 key、长 Lua 和无边界缓存。
- 账本、使用量、报表、审计四类数据分清主次，不要再把所有查询打到同一类存储。

### 安全措施

- 设定明确容灾目标，建议核心链路至少达到 `RPO <= 15 分钟`、`RTO <= 1 小时`。
- 日志、指标、Trace 三者必须可关联，事故时能追到 `request_id`、`user_id`、`api_key_id`、`account_id`。
- 对支付、计费、配额、退款、风控建立独立告警和独立值班预案。
- 定期做灾难恢复演练、越权审计、密钥轮换、漏洞应急演练。

### 人员配置

- 至少 `8-12` 人的后端/平台相关团队才比较稳。
- `1-2` 名专职 SRE/平台工程师基本是必需项，不再是可选项。
- 团队中需要有人长期负责数据库、中间件、成本治理和可观测性，而不只是功能开发。

### 其它关键动作

- 容量规划从“机器够不够”升级为“哪个资源先触顶、多久触顶、触顶时业务怎么降级”。
- 引入按租户、按模型、按地区、按上游账号的分层限流和配额治理。
- 建立正式的补偿体系，确保异步事件可重放、可追踪、可审计。

## 每个阶段都不能跳过的清单

### 备份与恢复

- PostgreSQL 每日备份，关键阶段升级为 PITR。
- Redis 明确持久化策略和恢复步骤。
- 每月至少做 1 次恢复演练，不能只有“理论上可恢复”。

### 监控与告警

- 至少监控：5xx、P95/P99、PostgreSQL 连接池、Redis 内存、磁盘使用率、支付 webhook 失败率、使用量写入延迟。
- 告警必须有明确接收人和处理时限，不能只是“发到群里”。

### 变更与回滚

- 所有发布都要有版本号、回滚方式、负责人。
- 支付、认证、计费相关变更不允许直接裸上生产。

### 安全

- 公网入口前放 CDN/WAF。
- 数据库、Redis、内部管理入口默认不暴露公网。
- 固定密钥、最小权限、2FA、审计日志是基础项，不是高级项。

## 常见误区

- 误区一：用户数刚到几百就上 Kubernetes。
  实际上这时更缺的是监控、备份、回滚和告警纪律。
- 误区二：把所有问题都归因于应用机器不够。
  很多时候先炸的是数据库、Redis、上游账号配额或异步任务积压。
- 误区三：为了扩容先拆微服务。
  对这个项目来说，很多时候先拆“控制面 / 数据面”比拆十几个服务更有效。
- 误区四：没有专人维护也能一直扛。
  当出现付费客户、夜间告警、补单退款和版本回滚压力后，专人维护就是收入保障。

## 推荐执行顺序

- 第一步：用仓库现有部署方案先把单机生产基线跑稳。
- 第二步：补监控、备份、回滚、安全基线。
- 第三步：做无状态横向扩容，优先把数据库迁托管。
- 第四步：治理 `usage_logs`、异步任务、支付与账务链路。
- 第五步：在 1 万到 5 万用户间完成控制面/数据面分离。
- 第六步：只有当多可用区、高 SLA、团队分工都成熟后，再上 Kubernetes 或更重的平台化方案。
